JAMstack

The JAM part of the achronym stands for - JavaScript - APIs - Markup We'll come back to this part later but first I want to draw your attention to this part of the definition...

"...Served Without Web Servers"

This sounds kinda crazy. [click]
It sounds like we're talking about a magical instance that runs websites without any servers, but that's not what this means.

Served Without Webservers

  • Web Apps/Websites are Hosted on a Global CDN
  • Not managing a dedicated Web Server(s)
  • Content is immediately distributed to the user
    • No server side rendering to generate dynamic content at runtime
  • What about server side API requests?
    • Serverless functions (Functions As A Service)
    • So long dedicated API Proxy!
Last point Doing this means content can be distributed immediately to the user when hosting via CDN.

JAMstack

JavaScript

  • Dynamic programming during the request/response cycle is handled by JavaScript
  • Frameworks: React, Angular, Vue etc
  • Vanilla JavaScript
  • Routing
  • UI Logic
  • Dynamic behaviour

API's

  • Server-side processes or database actions are abstracted into reusable APIs, accessed over HTTPS with JavaScript.
  • Leveraging third-party services: Fetching Content (Contentful/Sanity), Handling Payments (Stripe)
  • Custom-build services: User Authentication, Product Inventory

Markup

  • Templated markup should be prebuilt at deploy time, usually using a site generator for content sites, or a build tool for web apps.

Abstracting the Front End Layer gives us some added benefits

Better Performance

We have taken away the dependency of data at run time. Therefore we get what we need immediately. So we're still getting dynamic content, it's just the magic has happened before the user requests it. Making page load speeds much faster.

More Secure

There's reduced attack vectors. This is because we are serving pre-compiled HTML, with all the content we need, there's no direct path into the database via the front end layer. It's separate.
